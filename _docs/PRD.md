## 1. 产品概述

### 1.1 一句话定义
CoverFi 是预测市场 **Signa** 的本金保障层。用户在 Signa 下注后，可在 CoverFi 为该笔下注的**本金**投保；若该下注结算为「未命中」，投保人可取回 100% 本金，本金在 365 天内**线性释放**。

### 1.2 核心概念
- **本金 (principal / `a`)**：用户在 Signa 某笔订单中投入的金额。
- **保费 (premium)**：用户为投保支付的费用，动态计价（见 §3.2）。
- **保单 (policy)**：投保成功后生成的链上记录，绑定到投保人钱包，不可转让。
- **保障 (coverage)**：保单承诺的赔付额 = 100% 本金。
- **线性释放 (linear release)**：触发赔付后，本金在 365 天内按天匀速释放，用户可随时领取已释放部分。

### 1.3 链与环境
- **本次阶段：BSC Testnet（与 Signa 测试网同链）。**
- 后续阶段：BSC Mainnet（与 Signa 主网同链）。
- 测试网代币：BNB testnet（水龙头免费领取）。

### 1.4 非目标（本次不做）
- 主网上线、收取真实资金
- 保单转让 / 二级市场
- 多个预测市场支持（仅 Signa）
- 资金池的精算与偿付能力机制（见 §9 风险说明）

## 2. 用户流程

### 2.1 主流程
1. 用户已在 Signa 下注（前置条件，不在 CoverFi 内发生）。
2. 用户访问 CoverFi，连接钱包（与 Signa 同一钱包）。
3. CoverFi 通过 Signa 适配器读取该钱包**可投保的订单**。
4. 用户选择一笔订单 → 查看保费与条款 → 支付保费 → 铸造保单。
5. 等待 Signa 市场结算。
6. 若结算为「未命中」→ 保单进入「赔付中」，本金开始线性释放。
7. 用户随时回到「我的保单」领取已释放本金。

### 2.2 状态机（保单）
```
                    ┌─ 市场命中 ──────────→ [选项命中] (终态，保费留存)
[保障生效中] ──结算─┼─ 市场未命中 ─→ [赔付中] ──释放完成─→ [已赔付完成] (终态)
                    └─ 市场作废 ──────────→ [市场流局] (终态，保费退还)
```

**统一状态术语（全栈一致，不得更改措辞）：**
| 状态 key | 中文 | English |
|---|---|---|
| `active` | 保障生效中 | Coverage active |
| `releasing` | 赔付中 | Paying out |
| `completed` | 已赔付完成 | Reimbursed |
| `hit` | 选项命中 | Option won |
| `void` | 市场流局 | Market voided |

## 3. 业务规则（核心，需精确实现）

### 3.1 可投保条件
一笔 Signa 订单可被投保，当且仅当：
- 订单属于当前连接的钱包；
- 订单对应的市场在 CoverFi 白名单内；
- 市场尚未结算（仍开放）；
- 该订单尚未被投保过（一笔订单最多一份保单）。

### 3.2 保费计算
设本金为 `a`，隐含概率为 `k`，定价阀门为 `Q`。

**隐含概率 `k` 的定义：**
> `k = 用户所投选项的 TVL ÷ 该市场全部选项的总 TVL`
>
> 即：用户押注的那个选项上锁定的资金，占整个市场锁定资金的比例。
> 取值 0~1。两项数据（选项 TVL、市场总 TVL）均来自 Signa（见 §7）。

**保费公式：**
- 基础保费 = `Q × (1 − k) × a`
- 保费下限 = `0.05 × a`（本金的 5%）
- **实际应付保费 = max(基础保费, 保费下限)**

**定价阀门 `Q`：**
- `Q` 是一个**可由运营在后台调整**的全局参数（见 §4A 运营后台）。
- 默认值 `Q = 0.5`。
- `Q` 越大，整体保费越高；它是运营调节定价的核心旋钮。
- 保费计算时取**当前生效的 `Q` 值**；保单一旦生成，其保费即锁定快照。

**说明：** `k` 越高（市场越看好该选项命中），保费越低；反之越高。下限保证最低 5%。
保费与保障额在**购买时锁定快照**，此后市场变动、`Q` 变动均不影响已生成的保单。

> ⚠️ 计算精度：所有金额计算必须在代币最小单位（wei）上用整数运算，禁止用浮点数。前端展示时再做单位换算。

### 3.3 赔付与线性释放
- 仅当订单结算为「未命中」时触发赔付。
- 赔付总额 = 100% 本金 = `a`。
- 释放周期 = 365 天，自结算日起按天线性释放。
- 第 `d` 天已释放额 = `a × min(d / 365, 1)`。
- 可领取额 = 已释放额 − 已领取额。
- 用户可多次领取；未领取部分持续累积。

### 3.4 其他终态
- 「选项命中」：保险不触发，保单关闭，保费留存。
- 「市场流局」：保险不触发，保单关闭，**保费退还**给投保人。

## 4A. 运营后台（内部系统）

> 这是一个**独立于面向用户产品的内部管理系统**，仅供运营人员使用。
> 它有独立的页面、API 和权限控制，不对公众开放。

### 4A.1 目的
让运营人员无需改代码即可调整产品的关键参数。

### 4A.2 功能
| 功能 | 说明 |
|---|---|
| 定价阀门管理 | 查看 / 修改全局参数 `Q`（保费阀门，见 §3.2）。修改后即时对新报价生效。 |
| 市场白名单管理 | 增删 / 启用 / 停用可投保的市场（对应 `markets` 表的 `whitelisted`）。 |
| 保单总览（只读） | 查看全部保单、状态、金额，用于运营监控。 |
| 参数变更记录 | 记录每次 `Q` 或白名单的修改（操作人、时间、前后值），用于审计。 |

### 4A.3 访问控制
- 后台需登录鉴权（简单方案：固定管理员钱包地址白名单，或账号密码）。
- 普通用户无法访问后台路由与后台 API。
- 后台的"修改"类操作必须鉴权通过才执行。

### 4A.4 页面与端点

### 4A.5 数据表：`config`（全局参数）

## 6. 后端 API 规格

> RESTful，JSON。所有金额字段以 wei 字符串传输（避免精度丢失）。

| 方法 | 端点 | 说明 |
|---|---|---|
| GET | `/api/orders?address=` | 读取某钱包的可投保订单（经 Signa 适配器）|
| GET | `/api/orders/:id/quote` | 返回某订单的保费报价（按 §3.2）|
| GET | `/api/policies?address=` | 某钱包的全部保单 |
| GET | `/api/policies/:id` | 单张保单详情（含释放计算）|
| POST | `/api/policies` | 记录新铸造的保单（链上交易确认后）|
| POST | `/api/policies/:id/claim` | 记录一次领取 |
| GET | `/api/activities?address=` | 活动流水 |
| GET | `/api/markets` | 白名单市场列表 |

**注意：** 投保、领取的"真实动作"发生在链上（合约）；后端 API 负责索引链上事件、提供查询、计算释放进度。后端不托管资金。

## 8. 智能合约规格

> 语言 Solidity；部署目标 BSC Testnet；本次用 AI 审计 + 测试网验证。

### 8.1 合约：`CoverFiPolicy`
核心职责：
- `buyPolicy(orderId, ...)`：用户支付保费 → 铸造一份保单记录（保单数据上链）。
- `triggerSettlement(policyId)`：根据 Signa 结算结果，把保单转入 releasing/hit/void。
- `claim(policyId)`：投保人领取已线性释放的本金。
- 保单不可转让（不实现 transfer）。

### 8.2 关键约束
- 所有金额用 `uint256`，单位 wei。
- 线性释放按区块时间戳计算：`released = a * min(elapsed / 365days, 1)`。
- 资金来源：本次测试网阶段，合约持有的赔付资金由项目方在测试网预先注入（测试币，无成本）。**主网阶段的资金来源机制见 §9。**

### 8.3 安全要求（AI 审计须覆盖）
- 重入攻击防护（claim 函数）
- 整数溢出（Solidity ^0.8 默认带检查）
- 权限控制（谁能调 triggerSettlement）
- 校验 orderId 真实属于调用者、未被重复投保

## 9. ⚠️ 风险与未决事项（必须知晓）

> 以下不阻塞本次测试网开发，但**主网上线前必须解决**。

1. **偿付能力 / 资金来源未定义。** 用户付小额保费、却可赔付 100% 本金，差额来源未设计。测试网用预注入测试币绕过；**主网前必须设计资金池/精算/外部注资机制，否则系统在数学上无法持续兑付。** 文档作者已就此向产品方明确提示。
2. **未做专业合约审计。** 本次仅 AI 审计 + 测试网。主网前需专业审计。
3. **"保险"用语的合规性。** "保险/保单/理赔"在多数司法管辖区是受监管概念，主网/正式运营前建议法务确认。
4. **Signa 依赖。** 整个产品依赖 Signa 提供稳定的订单与结算数据；Signa 接口变更将直接影响 CoverFi。
5. **范围已扩大。** 运营后台（§4A）是原型未包含的新增模块，会增加约 2–3 小时工作量。若 20 小时见底，按 §4A.6 降级为最小版。
